# Bulk certificate extraction and translation with full-form edit fields

This change adds two admin-only bulk operations to the candidates list page—"Ekstraksi Semua Sertifikat" and "Translate Semua"—and expands the candidate edit form to expose every field imported from Form Responses 3. The bulk operations iterate over selected (or all filtered) candidates sequentially, calling the existing `/api/extract-cert-date` endpoint and the `translateToJapanese` client-side helper respectively, then writing results back to Firestore. The edit form grows from basic personal data to include reference/job info, health fields, passport/SIM, family (4 entries), education history (SD through university), work history (4 entries), Japan-related questions, emergency contacts, self-promotion, and document links.

**Watch for:** No confirmation dialog before kicking off potentially hundreds of sequential API calls (confirmed). Translation overwrites the entire `translations` object per candidate without merging existing keys that aren't in `TRANSLATABLE_FIELDS` (confirmed). The bulk extract loop makes up to 3 serial HTTP requests per candidate with no concurrency limit or abort mechanism—closing the browser tab mid-operation leaves partial writes in Firestore (confirmed).

## High-level view

The bulk extraction iterates all targets and fires up to three `/api/extract-cert-date` calls per candidate, one for each certificate URL field. There's no user confirmation, no abort button, and no error aggregation—failures are silently swallowed via `console.error`.

The bulk translation reads six motivational text fields per candidate, calls the MyMemory/Google fallback translation for each, then writes the merged `translations` object to Firestore. A 1-second delay between candidates provides rudimentary rate limiting, but no delay exists between the six fields within a single candidate—so six rapid-fire requests hit the free API in sequence.

The expanded edit form introduces `handleKeluargaChange` and `handlePekerjaanChange` helpers that default to a 4-element array of empty objects. If a candidate document has more than 4 family/work entries, the form silently truncates on save.

3. **No abort/cancel mechanism** — Once started, the only way to stop bulk processing is to navigate away or close the tab, leaving partial writes. Expose a cancel button that sets a ref flag checked each loop iteration.
4. **Silent failure accumulation** — Both bulk loops catch errors per-candidate and log to console. The user sees "done" even if 80% of extractions failed. Surface a summary count of successes/failures after completion.

## Bulk extraction: sequential OCR with no guardrails

The `handleBulkExtract` function processes candidates one-by-one. For each candidate it conditionally calls `/api/extract-cert-date` for `sertifikatBahasaJepang`, `sertifikatSSW`, and `sertifikatSenmonkyuu`. Each call downloads a PDF from Google Drive, parses text or falls back to OCR via the OCR.space API. The three calls are sequential—a candidate with all three URLs generates three round-trips before the loop advances.

The target selection logic falls through to `filtered` when `selected` is empty:

```js
const targets = selected.length > 0
  ? filtered.filter((c) => selected.includes(c.id))
  : filtered;
```

This means clicking the button with nothing selected processes every visible candidate—potentially the entire database if no filter is active. Combined with no confirmation dialog, a mis-click can trigger hundreds of OCR calls against a quota-limited API (OCR.space free tier is 25,000/month).

There is no way to cancel mid-run. The progress banner shows "N/M" counts but doesn't surface individual failures. A user may watch it complete and assume success when most calls returned 422 errors for private or non-PDF files.

## Bulk translation: object merge semantics

```js
const translations = candidate.translations || {};
for (const field of TRANSLATABLE_FIELDS) {
  if (candidate[field] && candidate[field].trim()) {
    const translated = await translateToJapanese(candidate[field]);
    translations[field] = translated;
  }
}
await updateDoc(doc(db, "candidates", candidate.id), { translations });
```

The `updateDoc` call with `{ translations }` replaces the entire `translations` map in Firestore with the local object. Because it starts from `candidate.translations || {}`, existing keys outside `TRANSLATABLE_FIELDS` are preserved—but only because the candidate data was freshly loaded. If another user or process wrote a new translation key between `loadCandidates()` and the write, that key is lost. This is a last-write-wins race without optimistic locking.

The six per-candidate translation calls hit MyMemory's API with no inter-field delay. MyMemory's free tier documents a 5-request/second limit; six sequential awaits may not breach it given network latency, but it's borderline.

## Edit form: nested array handling

The new `handleKeluargaChange` and `handlePekerjaanChange` helpers clone (via spread) the existing array or default to `[{}, {}, {}, {}]`. If a document was imported with 5 or 6 family entries, the form only renders indices 0–3 (hardcoded in JSX as `[0, 1, 2, 3].map(...)`), and the default 4-element array would silently truncate extra entries on save because `handleSave` spreads `...data` which now contains the trimmed array.

Fields like "Buta Warna", "Merokok", "Tato" are yes/no values but render as free-text inputs, creating inconsistent data shapes across candidates depending on who enters the data.
